구닥다리 공룡을 위한 모던 자바스크립트

Frontend 개발자는 HTML, CSS, 그리고 JavaScript만 있는면 되는거 아니었어?
근데 node, npm, grunt, gulp, webpack, babel은 또 뭐야!
저것들이 왜 필요한거지?

물론 지금도 HTML, CSS, 그리고 JavaScript만 있으면 개발이 가능합니다만

  • 빠르게 성장하는 생태계
  • 이로인해 생겨난 문제들

‘문제’를 해결하기 위해 도구들이 등장하기 시작합니다. 여기서는 JavaScript 중심으로 현재까지 진화해온 도구의 역사를 정리해봅니다.

구식 JavaScript 사용법

HTML과 JavaScript를 사용한 웹페이지 하나를 만들어보겠습니다.

<!-- index.html -->
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <title>웹 페이지</title>
</head>
<body>
  <h1 id="hello">안녕! 나는 웹 페이지야.</h1>
  <script src="index.js"></script>
</body>
</html>

<script src="index.js"></script> 이 부분에서 index.html과 같은 폴더에 위치한 index.js를 참조합니다.

/* index.js */
console.log('안녕! 나는 콘솔에 로그를 찍어주지.');

이게 다에요. 뭔가 외부 라이브러리를 추가하고싶다면 다음과 같이 추가하세요.

<!-- index.html -->
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <title>웹 페이지</title>
</head>
<body>
  <h1 id="hello">안녕! 나는 웹 페이지야.</h1>
  <script src="jquery-3.3.1.min.js"></script>
  <script src="index.js"></script>
</body>
</html>

잘 찾아보세요. jquery를 추가한 것을 볼 수 있습니다. index.js보다 먼저 로드되어 index.js에서 jquery 특성을 사용할 수 있게됩니다.

console.log('안녕! 나는 콘솔에 로그를 찍어주지.');
$('#hello').text('반가워! 나는 jquery야.');

이게 구닥다리 방식입니다. 이해하기 쉽지만 단점이 하나있는데 바로!

라이브러리 업데이트가 있으면 새로운 버전을 찾아서 교체해줘야 하니 라이브러리 수가 많아지면 곤란하겠군

패키지 매니저의 등장

2010년 즈음, 중앙 저장소로부터 라이브러리를 다운받고 업데이트를 자동으로 해주는 패키지 매니저들이 등장합니다.

  • 2013년, Bower가 최고의 인기 제품 이었습니다.
  • 2015년 즈음, npm이 Bower의 인기를 따라잡습니다.
  • 2016년 말, yarn이 npm의 대안으로 등장합니다.(내부적으로는 npm을 사용한다고 하네요.)

npm은 원래 node.js를 위해 특별히 만들어진 것으로 frontend가 아닌 server 위에서 구동되도록 설게되었습니다. 그러고보면 npm을 frontend 패키지 매니저로 선택한 것이 특이한 점이네요. 아무튼! npm으로 어떻게 패키지 관리가 이루어지는지 알아보겠습니다.

  1. 먼저 node.js를 설치합니다.(npm도 같이 설치됩니다.)
  2. index.html을 작성한 폴더로 이동하여 다음 명령어를 입력합니다.
    $ npm init
  3. 몇가지 질의응답 과정이 이어집니다. 나중에 수정이 가능하므로 모든 질문에 엔터를 눌러 진행합니다.
  4. package.json이라는 설정 파일이 하나 생성됩니다. 이 파일에 프로젝트에 대한 모든 정보를 저장하며 내용은 아래와 같습니다.
    {
"name": "example",
"version": "1.0.0",
"description": "",
"main": "index.js",
"scripts": {
 "test": "echo \"Error: no test specified\" && exit 1"
},
"author": "",
"license": "ISC"
}
  5. 자! 이제 jquery를 npm으로 설치할 수 있게됩니다. 다음 명령어를 입력하여 설치하세요.
    $ npm install jquery --save
  6. 위 명령이 하는 일은 첫째로 jquery의 모든 코드를 다운받아 node_modules라는 폴더에 저장하며 둘째로 package.json 파일을 아래와 같이 고쳐서 프로젝트의 의존성으로서 jquery를 지속적으로 추적하도록 합니다.
    {
"name": "example",
"version": "1.0.0",
"description": "",
"main": "index.js",
"scripts": {
 "test": "echo \"Error: no test specified\" && exit 1"
},
"author": "",
"license": "ISC",
"dependencies": {
 "jquery": "^3.3.1"
}
}

이는 나중에 다른 이와 프로젝트를 공유할 때 유용합니다. node_modules 폴더를 직접 공유할 필요없이 package.js 하나만 공유하여 npm install 명령을 수행해주면 되기 때문이죠. node_modules 폴더 안을 보면 jquery파일이 node_modules/jquery/dist/jquery.js 폴더 안에 있습니다. 이제 우리는 npm이 다운로드한 jquery를 index.html에 연결할 수 있습니다.

<!-- index.html -->
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <title>웹 페이지</title>
</head>
<body>
  <h1 id="hello">안녕! 나는 웹 페이지야.</h1>
  <script src="node_modules/jquery/dist/jquery.min.js"></script>
  <script src="index.js"></script>
</body>
</html>

자 이제 우리는 command line 환경에서 npm을 사용하여 package들을 다운로드하고 업데이트할 수 있습니다. 하지만 아직도 불편한 건 바로!

라이브러리를 연겷하는데 ‘node_modules/jquery/dist/jquery.min.js’와 같이 경로를 일일이 다 적어줘야하나

JavaScript 모듈 번들러 ‘Webpack’ 사용하기

대부분의 프로그래밍 언어는 하나의 코드에서 다른 코드를 불러들이는 방법을 제공합니다만 JavaScript는 없습니다. 그 이유는 바로

  • JavaScript가 오직 브라우저 환경에서만 구동되도록 만들어졌고
  • 보안상의 이유로 로컬의 파일 시스템의 접근할 수 없기 때문

때문에 지금까지 HTML파일에서 연결을 했던 것이었습니다. 이렇게 연결된 라이브러리는 이후 연결되는 모든 파일에서 전역변수로 사용됩니다. 그러던 중 변화가 일어나기 시작합니다.

2009년, browser외의 JavaScript 생태계를 만들기 위한 CommonJS라는 이름의 프로젝트 시작

이 프로젝트는 대부분 모듈에 대한 명세였고 이 명세로 마침내 JavaScript도 하나의 코드에서 다른 코드를 들여오고 내보낼 수 있게 됩니다. 이러한 모듈을 구현한 것으로 node.js가 있습니다. 아래 초기 node.js 모듈의 사용 예시를 확인해보세요.

// index.js
var $ = require('jquery');
console.log('안녕! 나는 콘솔에 로그를 찍어주지.');
$('#hello').text('반가워! 나는 jquery야.');

이것이 바로 node.js의 모듈 불러오기 방식입니다. node.js가 파일 시스템에 접근 가능한 서버사이드 언어이기 때문에 가능한 것이지요. node.js는 각 모듈이 어느 경로에 있는지 알기 때문에 require('jquery')라고 쓸 수 있는 것입니다.

node.js에서는 이렇게 할 수 있다고 하지만 브라우저 상에서 실행하려고 하면 require가 정의되지 않았다고 에러를 뿜어냅니다. 또한 브라우저는 파일 시스템에 접근할 권한이 없기도 합니다. 여기서 모듈 번들러가 필요해지기 시작합니다. JavaScript 모듈 번들러는 파일 시스템에 접근할 수 있는 빌드 단계에서 브라우저와 호환되는 최종 결과물을 만들어냅니다. 간단하게 설명하자면 다음과 같습니다.

  • require 구문을 찾아내어
  • 불러오는 모듈의 실제 내용으로 대치시켜준다.
  • 이때 최종 결과물은 하나로 묶인(bundle) 단일 JavaScript 파일이 된다.

가장 인기있었던 모듈 번들러는 2011년에 등장한 Browserify 였으며, npm이 frontend 패키지 매니저가 될 수 있게 해주었습니다. 2015년 즈음엔 webpack이 등장하여 보편적으로 사용되는 모듈 번들러가 되었습니다. 자! 이제 webpack을 설치하고 사용해 보겠습니다.

  1. 설치
    npm install webpack webpack-cli --save-dev
    개발환경에서만 사용되며 실제 배포이후에 사용되지 않을 경우 --save-dev 옵션을 주어 명령을 실행합니다. package.json 에는 다음과 같이 들어가게 됩니다.
    {
"name": "example",
"version": "1.0.0",
"description": "",
"main": "index.js",
"scripts": {
 "test": "echo \"Error: no test specified\" && exit 1"
},
"author": "",
"license": "ISC",
"dependencies": {
 "jquery": "^3.3.1"
},
"devDependencies": {
 "webpack": "^4.4.1",
 "webpack-cli": "^2.0.13"
}
}
  2. 최종 결과물 만들기
    이제 다음 명령을 입력하여 최종 결과물을 만들어봅시다.
    $ ./node_modules/.bin/webpack index.js
    명령을 실행하고 나면 dist 폴더와 함께 main.js 파일이 생성됩니다. index.js 파일에 jquery의 내용을 합쳐서 하나의 파일을 만들어 낸 것입니다.
  3. 결과물(main.js)를 연결하기
    이제 main.js 하나면 됩니다. 아래와 같이 연결해줍니다.
    <!-- index.html -->
<!DOCTYPE html>
<html lang="ko">
<head>
<meta charset="UTF-8">
<title>웹 페이지</title>
</head>
<body>
<h1 id="hello">안녕! 나는 웹 페이지야.</h1>
<script src="dist/main.js"></script>
</body>
</html>

사용하는 라이브러리가 많아도 이렇게 하나로 만들어주면 일일이 연결해 줄 필요가 없어 편리해집니다. 하지만 아직도 불편한 것들이 존재합니다.

  • index.js가 수정되면 webpack 명령을 다시 수행해야하네
  • webpack의 고급기능을 사용하게 되면 어떻게 되는거지?

이런 문제들을 해결하기 위해 webpack은 config 파일을 사용합니다. 프로젝트의 루트 폴더에 webpack.config.js를 생성하고 다음과 같이 작성하세요.

// webpack.config.js
module.exports = {
  entry: './index.js',
  output: {
    filename: 'dist/main.js'
  }
};

이제 index.js가 바뀔 때 마다 조금은 간소해진(index.js를 명시하지 않아도 되요 ^^) 아래 명령을 수행하면 됩니다.

$ ./node_modules/.bin/webpack

이제 더이상 HTML 파일에서 JavaScript 파일을 연결하여 전역변수로 불러들일 필요가 없어졌습니다. require를 통해 JavaScript 파일을 불러오면 되는 것입니다. 많이 편해졌지만 여전히 귀찮은 것들을 해결해 나가보도록 하겠습니다.

JavaScript의 새로운 기능을 사용하기 위한 code transpiler ‘Babel’

transpile 한다는 것은 어떤 언어를 다른 언어로 변환시켜 준다는 것입니다. 브라우저가 ‘새로운 언어’를 지원하는데 항상 늦장을 부리기 때문에 transpiler가 필요해지게 되었습니다. 그렇다면 ‘새로운 언어’가 어떤 것들이 있을까요?

  • Sass, Less, Stylus
  • CoffeeScript, Babel, TypeScript

우리는 CSS가 아닌 JavaScript에 집중하기 때문에 그 중에서 많이 쓰이는 Babel을 알아보겠습니다. 알다시피 Babel은 아직 모든 브라우저에서 사용할 수 없는 ES2015 이후의 기능을 오래되고 호환 가능한 JavaScript로 변환해주는 transpiler 입니다. 이제 Babel을 설치하고 사용해보도록 하겠습니다.

  1. npm을 통해 설치
    $ npm install --save-dev babel-core babel-preset-env babel-loader
    참고로 core는 말 그대로 핵심, preset-env는 어떠한 JavaScript 새 기능을 변환할지에 대한 사전 정의입니다. loader는 Babel이 webpack과 함께 일할 수 있도록 해주는 패키지입니다.
  2. webpack.config.js 파일 수정
    // webpack.config.js
module.exports = {
entry: './index.js',
output: {
 filename: 'main.js'
},
module: {
 rules: [
   {
     test: /\.js$/,
     exclude: /node_modules/,
     use: {
       loader: 'babel-loader',
       options: {
         presets: ['env']
       }
     }
   }
 ]
}
};
    webpack에게 node_modules 폴더를 제외한 모든 JavaScript 파일들을 지켜보도록 합니다. 그리고 babel-loader를 통해 변환을 적용합니다.
  3. index.js 파일을 수정
    // index.js
import $ from 'jquery';
console.log('안녕! 나는 콘솔에 로그를 찍어주지.');
var hello = "반가워!", name = "Jason";
$('#hello').text(`${hello} 나는 ${name}이라고 해.`);
    자 이제 JavaScript의 새로운 기능들을 사용할 수 있습니다.
    ES2015 template 문자열의 예시를 추가하였습니다. 또한 require 대신 ES2015 import 구문을 사용하도록 바꾸겠습니다.
  4. webpack 명령어 실행
    $ ./node_modules/.bin/webpack
    브라우저에서 index.html를 새로고침하여 확인해 보세요. 특히 IE9과 같은 구식 브라우저에서 확인해보시길 바랍니다.

transpiler는 이렇게 우리가 더 좋은 코드를 작성할 수 있도록 해줍니다. 이제 거의 다왔습니다. 다음 두 가지가 바로 그것입니다.

  1. 성능을 고려해 본다면 우리가 webpack을 통해 만든 bundle 파일을 축소시킬 필요가 있을 것입니다.
  2. JavaScript가 변경될 때마다 webpack 명령을 수행해 주어야 하는 건 꽤 귀찮습니다.

task runner 사용하기

task runner란 각기 다른 부분의 빌드 과정을 자동화 해주는 도구입니다. Frontend 개발에 있어 task란 다음을 말합니다.

  • 코드 압축
  • 이미지 최적화
  • 테스트 실행

Grunt와 Gulp가 있지만 여기서는 npm script를 활용해보겠습니다. 이를 위해 package.json을 수정하도록 하겠습니다. scripts에 build와 watch명령을 추가하겠습니다.

{
  "name": "example",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "build": "webpack --progress -p",
    "watch": "webpack --progress --watch"
  },
  "author": "",
  "license": "ISC",
  "dependencies": {
    "jquery": "^3.3.1"
  },
  "devDependencies": {
    "babel-core": "^6.26.0",
    "babel-loader": "^7.1.4",
    "babel-preset-env": "^1.6.1",
    "webpack": "^4.4.1",
    "webpack-cli": "^2.0.13"
  }
}

추가한 명령을 수행해 보겠습니다. 빌드는 아래와 같이 입력합니다.

$ npm run build

참고로 --progress 옵션은 진행 과정을 백분율로 보여주며 -p는 코드를 압축해줍니다. watch는 아래와 같이 입력합니다.

$ npm run watch

--watch는 JavaScript가 수정될 때마다 자동으로 webpack을 재실행 해줍니다.

이렇게 앞서 언급했던 두 가지를 해결했습니다. 또한 이전에 webpack 명령을 수행하려면 경로를 적어주어야 했던 부분도 깔끔하게 정리되었습니다. 여기에 webpack-dev-server를 설치하면 더 좋아집니다. 실시간 다시 불러오기(Live reloading)기능과 웹 서버까지 제공해주기 때문입니다. 설치는 npm을 통해 이루어집니다.

$ npm install webpack-dev-server --save-dev

package.json 파일에 서버 실행을 위한 스크립트를 추가하면 아래와 같습니다.

{
  "name": "example",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "build": "webpack --progress -p",
    "watch": "webpack --progress --watch",
    "server": "webpack-dev-server --open"
  },
  "author": "",
  "license": "ISC",
  "dependencies": {
    "jquery": "^3.3.1"
  },
  "devDependencies": {
    "babel-core": "^6.26.0",
    "babel-loader": "^7.1.4",
    "babel-preset-env": "^1.6.1",
    "webpack": "^4.4.1",
    "webpack-cli": "^2.0.13",
    "webpack-dev-server": "^3.1.1"
  }
}

자 이제 실행해 보겠습니다.

$ npm run server

실행하면 자동으로 웹 브라우저가 열립니다. 기본 포트는 8080입니다. 언제든 자바스크립트를 수정하면 webpack-dev-server는 번들된 JavaScript를 다시 빌드하고 브라우저를 새로고침합니다. 이는 놀라울 정도로 시간을 절약해주며 계속 코드에 집중할 수 있게끔 해줍니다.

이렇게만 해도 좋은데 여기까지는 아직 수박 겉 핡기 수준이라고 합니다. 훨씬 많은 옵션들이 webpack과 webpack-dev-server에 존재하며 다른 task를 위해 스크립트를 작성할 수 있습니다.

  • Sass를 CSS로 변환
  • 이미지 압축
  • 테스트 실행

이밖에 여러가지 일들을 npm script를 통해 수행할 수 있습니다.

결론

간단히 말하자면

이것이 바로 모던 JavaScript이다.

계속되는 변화와 급격한 진화 속도로 인해 진통을 겪을지는 모르지만 분명한 것은 이러한 것들을 개발자들이 코드에 집중하게 만드는 혁신을 이루고 있다는 것입니다.
